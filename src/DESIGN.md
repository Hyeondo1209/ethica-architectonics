## 0. 완결의 정의 / 방법론 / 역할
- **완결 = 에티카 1부(정리 1~15)의 웹 3D 건축 번역.** 5부 전체 아님. **깊이 우선**(끝까지 구현된 1부 > 얕은 5부). 16~36(사물의 생산)은 욕심 목표.
- **방법론:** 마르샬 게루의 발생적-구조적 독해. 신을 *전제*하지 않고, 단일 속성 실체에서 신 개념을 *구성*해 나가는 과정으로 1부를 파악. (들뢰즈 비교, 내들러 보조.)
- **역할:** 철학적 해석의 최종 판단·미적 결정 = **사용자(김현도)**. Claude = 구조화·구현·비판적 검토. **사용자의 독해를 멋대로 바꾸지 않는다.**
- **스택(확정):** React Three Fiber(three.js). 주연 에셋만 Blender→glTF. 배포 Vercel. **합의 없이 스택 변경 금지.** 반복·절차적 구조는 코드로 생성(손모델링 X).

## 1. 못 박은 원칙 (LOCKED — 절대 잊지 말 것)
- ★★ **단일 속성 실체 = 항상 얇고 우아하게 휜(S자 / 탄젠트 곡선) "경선 리브" 그 자체.** 두꺼운 별도 기둥이 아니다. 리브 1개 = 단일 속성 실체 1개. 여럿(≈72)이 무한 반복을 암시.
- ★★ **탐험 = 그 경선 리브 *중 하나* 안의 나선을 오르는 것.** 별도 종류의 구조물이 아니라 한 리브의 *역할*일 뿐. → **탐험 리브를 위치·굵기·형태로 차별화하면 안 된다**(전부 하나의 실체이므로). 부득이한 차별(들어갈 굵기)은 *공간 전체를 키워서* 흡수한다.
- **동선:** 지하 정의·공리 방(중앙) → 공중 통로(정리 1~4) → 가장자리 탐험 리브 → 리브 안 나선 → 테라스(도착). **중앙에서 곧장 위가 아니라, 중앙(지하) → 수평 바깥 → 가장자리에서 위.**
- **돔 = 내재성('외부 없음').** 정의·공리 = **지하**. 정리 1~4 = **공중 통로 위**. (이전 "정의·공리 바닥 배치 검토 중", "외부 없음 미해결"은 모두 *해결됨* — 낡은 `작업베이스.md` 무시.)

## 3. 공간 모델 + 현재 파라미터 (현 코드 기준)
- **돔/리브:** `R_BASE=60`(지표) → `R_TOP=35`(천장 개구부), `H=200`, 무릎 `KNEE=0.25`(y50)에서 tanh로 급히 좁아짐. `MERIDIANS=72`, 리브 굵기 `SHELL_RIB_R=1.1`(얇음). **선호 밀도 = 직전 스크린샷의 빽빽한 72개 느낌.**
- **탐험(현 상태, 전환 전):** `ClimbShaft`(반투명, `SHAFT_R=5`) at 반지름 `ARRIVAL_R=47.5`(+x). 나선 `RibStair`(`STAIR_R=2.8`, `STAIR_TURNS=4.5`) → `Y_LAND=47`. → `LandingPad`(`PAD_CX=40.2`)·`SmallLandingPad` → `StraightFlight`(8칸, 안쪽으로) → 테라스.
- **테라스:** 중앙 부분 링 `TERRACE_RIN=27`~`TERRACE_ROUT=33`, y50, 호 `TERRACE_ARC=2.4`(+x 향함).
- **정점:** `Apex` y200, 광원+구.
- **지하 방:** 묻힌 돔 `ROOM_R=22`, 중심 `ROOM_CY=-18`(top y4, 통로 아래), 바닥 `ROOM_FLOOR_Y=-14`. 정의 링 `DEF_RING_R=7`(마커 8)·공리 링 `AXIOM_RING_R=12`(마커 7), `MARKER_R=0.9`. 정의=안쪽/따뜻한색, 공리=바깥/차가운색.
- **공중 통로:** 타원 platform y `WALK_Y=5`, 중심 x `WALK_CX=25`, 반축 `WALK_RX=30`×`WALK_RZ=14.5`. (정리 1~4 마커는 임시 제거됨 — 추후 번호 라벨.)
- **사람 스케일(고정):** 눈높이 `EYE=1.6`, 속도 `SPEED=4.5`, 턱 `STEP_UP=0.8`/`STEP_DOWN=2.2`, 낙하 `FALL=5`. **스케일업 시 이 값들은 고정**, 건축 치수만 키운다.
- **조작/임시:** `FREE_WALK=true`(벽·가장자리·낙하 충돌 끔 — 편집용 임시. 끄려면 false). `G`키로 돔 씬↔데이터 그래프 뷰.

## 5. 진행 중인 큰 전환 (NEXT SESSION의 핵심 작업)
**"두꺼운 기둥 → 얇은 경선 리브 = 실체"로의 전환 + 공간 스케일업.**
1. `Pillars`(두꺼운 원기둥) **삭제**.
2. 단일 속성 실체는 **`DomeRibs`(얇은 경선 리브) 자체**임을 코드/주석에 반영.
3. 나선은 **경선 리브 *중 하나* 안**에 둔다. 리브의 **곧은 아래 구간**(무릎 아래, 지표~y≈40에서 거의 수직)에 나선을 넣어 무릎(테라스)에 닿게 — 휘는 구간을 따라갈 필요 없음.
4. **공간 전체 스케일업**(사람 치수 고정). 둘레가 커져야 "사람이 들어갈 만큼 굵지만 전체 대비 여전히 얇은" 리브가 가능. 스케일 배수를 건축 치수(돔·방·통로·테라스·나선 반지름)에 곱하고, 안개·라이트·이동거리도 비례 조정.
   - 트레이드오프: **리브 수↓(예 72→36)면 같은 굵기라도 덜 빽빽** → 덜 키워도 얇아 보임. 단 사용자는 *직전 스크린샷의 빽빽한 72개 밀도 선호* → 밀도 유지하며 스케일로 푸는 방향.
5. 스케일 정하면 파라미터로 조율(빽빽함·굵기·이동감).

## 6. 열린 / 미룬 결정
- 스케일 배수 확정(위 전환) — 밀도(72 유지) vs 스케일 균형.
- 관람 시작 위치를 방/통로 안쪽으로(현재 등반 기둥 밑).
- 테라스 위치(중앙 유지 — 확정 ㉮) / 탐험 리브 top과의 연결 매끈하게.
- 공중 통로 높이(y5) vs 나선 바닥(y0) 틈 보정.
- 번호 라벨(1d/1a/1p) 표시 방식.
- 정의·공리 방 지표 입구·하강 동선.
- 9~15 의존 데이터 입력(형태 먼저, 데이터 뒤 — 보류).
- drei 도입 여부. `FREE_WALK` 임시(편집 끝나면 false).
- 16~36 확장 여부(욕심 목표).
